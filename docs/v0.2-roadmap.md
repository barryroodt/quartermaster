# v0.2 Roadmap

## MCP tool enumeration

The MCP fetcher in `src/cli.ts` is stubbed to return zero tools. Wiring real MCP tool discovery requires:

- An MCP stdio client (the `@modelcontextprotocol/sdk` package or a hand-rolled JSON-RPC client over stdio)
- Spawning each configured MCP server, calling `tools/list`, parsing the response into `McpTool[]`
- Handling server startup failures gracefully (the inventory should not block on a broken server)

Until wired, users with MCP servers configured see a warning at `/qm init` time and zero `mcp_tool` rows in their inventory.

## /qm install for skill:raw and mcp:* capability ids

`runInstall` (src/commands/install.ts) dispatches `skill:skills-sh:*`, `cli:brew:*`, and `plugin:claude:*` shapes. The `skill:raw:*` and `mcp:*` shapes return a "failed" outcome with a v0.2 marker. Wiring requires:

- For `skill:raw:*`: parse the capability_id into url + slug pair, call `installSkillRaw`
- For `mcp:*`: accept transport args via `--transport-arg=<arg>` flags (already parsed by cli.ts), call `installMcp`. Needs UX work for how transport args are surfaced to the user during gap-search (currently gap-search only emits skills.sh + brew hits)

## EnterPlanMode handoff

`/qm survey` emits `next: /plan "<goal>"` as a text suggestion. Direct invocation of Claude Code's EnterPlanMode tool from a Bun subprocess is not possible (subprocesses run outside the agent's tool context). True handoff requires either:

- A slash-command shim that runs `/qm survey` then `/plan` sequentially
- A structured marker in the survey output that an enclosing skill picks up

## Interactive prompt handling

The installer (src/installer/prompts.ts) defines `formatUntrustedPrompt`, `formatDriftPrompt`, `formatPromotePrompt`. v0.1 uses non-interactive `--yes` / `--yes-drift` flags instead of TTY readline (Bun subprocess has no usable stdin from the agent). Real interactive prompts require wrapping the CLI in a parent shell that owns the TTY.
