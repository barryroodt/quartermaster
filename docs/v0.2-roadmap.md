# v0.2 Roadmap

## /qm install for skill:raw

`runInstall` dispatches `skill:skills-sh:*`, `cli:brew:*`, `plugin:claude:*`, and `mcp:*` shapes. The `skill:raw:*` shape returns a "failed" outcome with a v0.2 marker. Wiring requires parsing the capability_id into url + slug pair, then calling `installSkillRaw`. Low effort — held back only by gap-search not currently emitting raw-skill hits (no registry produces them; would need URL-paste UX).

## EnterPlanMode handoff

`/qm survey` emits `next: /plan "<goal>"` as a text suggestion. Direct invocation of Claude Code's EnterPlanMode tool from a Bun subprocess is not possible (subprocesses run outside the agent's tool context). True handoff requires either:

- A slash-command shim that runs `/qm survey` then `/plan` sequentially
- A structured marker in the survey output that an enclosing skill picks up

## Interactive prompt handling

The installer (src/installer/prompts.ts) defines `formatUntrustedPrompt`, `formatDriftPrompt`, `formatPromotePrompt`. v0.1 uses non-interactive `--yes` / `--yes-drift` flags instead of TTY readline (Bun subprocess has no usable stdin from the agent). Real interactive prompts require wrapping the CLI in a parent shell that owns the TTY.

## MCP tool-level enumeration

The `mcpFetcher` in `src/cli.ts` is stubbed to return zero tools per server. **This is correct by design.** Per spec line 22, individual MCP tools are deferred and only surfaced via Claude Code's built-in ToolSearch. The `mcp_tool:*` rows in the schema exist for future use if tool-level discovery becomes valuable (e.g. for proactive tool-availability nudges), but are not v0.2 scope.

## MCP trust resolution

`/qm install mcp:*` currently requires `--yes` uniformly because MCP server names don't carry a github source_url. v0.2 could add a name-to-repo mapping (e.g. `@modelcontextprotocol/server-foo` → `github.com/modelcontextprotocol/servers`) so trusted org patterns auto-approve.
